# WHOOP Data Pipeline & Dashboard

Personal biometric data pipeline: WHOOP API → OAuth 2.0 → SQLite → Streamlit dashboard.
Built as a co-op portfolio project.

## File structure

```
whoop-pipeline/
├── .env                        # your real credentials (gitignored, never committed)
├── .env.example                # template showing required env vars
├── .gitignore
├── README.md
├── requirements.txt
├── whoop_auth.py                # OAuth 2.0 flow -> produces tokens.json
├── pull_data.py                 # pulls cycles/recovery/sleep/workouts -> whoop.db
├── validate.py                  # data quality checks on whoop.db
├── dashboard.py                 # Streamlit app reading from whoop.db
├── tokens.json                  # generated by whoop_auth.py (gitignored)
├── whoop.db                     # generated by pull_data.py (commit this once
│                                 # you want data in the repo, or gitignore it
│                                 # and let GitHub Actions maintain it)
├── .github/
│   └── workflows/
│       └── daily_pull.yml       # scheduled daily pull + validation + commit
└── docs/
    └── index.md                 # Privacy Policy page, served via GitHub Pages
```

**Note on `whoop.db`:** it's listed in `.gitignore` right now (personal
health data). If you want the dashboard to work when someone views your
repo, either (a) remove it from `.gitignore` and commit it deliberately
once you're comfortable with that data being public, or (b) keep it
private and rely on a screenshot/GIF in the README instead. Your call —
flagging it so it's a decision, not an accident.

## 5-Day Build Plan

- **Day 1 — Auth + first pull**
  - Create app at developer.whoop.com/dashboard, set redirect URI to `http://localhost:8000/callback`
  - Copy `.env.example` → `.env`, fill in your Client ID/Secret
  - `pip install -r requirements.txt`
  - `python3 whoop_auth.py` — authorizes in browser, saves `tokens.json`
  - Commit: auth script + README (not `.env`/`tokens.json` — they're gitignored)

- **Day 2 — Ingestion + schema**
  - `python3 pull_data.py` — pulls cycles/recovery/sleep/workouts into `whoop.db`
  - Inspect the schema, tweak table design if you want more normalization
  - Commit: pipeline script, schema notes

- **Day 3 — Automate + validate**
  - Set up a scheduled pull: cron (`crontab -e`, run `pull_data.py` daily) or a
    GitHub Actions workflow on a schedule
  - Add 2-3 sanity checks (e.g., recovery_score between 0-100, no duplicate cycle ids)
  - Commit: automation config, validation script

- **Day 4 — Dashboard**
  - `streamlit run dashboard.py`
  - Recovery trend, strain-vs-recovery scatter, HRV trend, sleep performance
  - Commit: dashboard code + a screenshot in the README

- **Day 5 — Polish + tell the story**
  - Write the case-study section below with your actual numbers
  - Record a short screen capture/GIF of the dashboard
  - LinkedIn post: architecture diagram + one real insight from your own data
  - Final commits: polish, docs, tag a `v1` release

## Architecture

```
                                            |
                                            v
                                     pull_data.py
                                            |
                                            v
                                       whoop.db (SQLite)
                                            |
                                            v
                                      dashboard.py (Streamlit)
```

## Honest scope note
Early on you'll have very few days of data — trend lines and correlations
won't be statistically meaningful yet. Say so explicitly in your writeup;
it reads as rigor, not weakness. The pipeline and automation are the real
engineering story in week one; the data-driven insights get more
interesting as days accumulate.
